# 1.2.3 System Specifications

Translating sentences from natural language into logical expressions is an essential part of specifying both hardware and software systems. System engineers and software engineers take requirements written in natural language (like English) and produce precise, completely unambiguous specifications that can serve as the baseline for system development.

### Consistency in System Specifications

A critical goal in engineering is to ensure that a set of system specifications is **consistent**.

> **Definition:** A list of system specifications is **consistent** if and only if there is an assignment of truth values to the propositional variables that makes **all** the specifications true simultaneously.

If a set of specifications is *not* consistent, it contains conflicting requirements. This means there is a logical contradiction hidden in the rules, making it physically impossible to build a system or write software that satisfies all the specifications at the same time.

### Textbook Walkthroughs: Checking for Consistency

#### EXAMPLE 3: Translating a Single Rule

> **Specification:** “The automated reply cannot be sent when the file system is full.”

1. **Define the variables:**
   * Let $p$ denote: “The automated reply can be sent.”
   * Let $q$ denote: “The file system is full.”
2. **Translate components:**
   * “The automated reply cannot be sent” becomes $\neg p$.
   * The word "when" acts as an indicator that the condition is the hypothesis. If the file system is full ($q$), then the outcome triggers ($\neg p$).

$$\text{Final Expression: } q \rightarrow \neg p$$

---

#### EXAMPLE 4: Retransmission Buffer Consistency

> Determine if these three specifications are consistent:
> 1. “The diagnostic message is stored in the buffer or it is retransmitted.”
> 2. “The diagnostic message is not stored in the buffer.”
> 3. “If the diagnostic message is stored in the buffer, then it is retransmitted.”

**Step 1: Translate into logical expressions**
* Let $p$ denote: “The diagnostic message is stored in the buffer.”
* Let $q$ denote: “The diagnostic message is retransmitted.”

Our three requirements translate to:
1. $p \lor q$
2. $\neg p$
3. $p \rightarrow q$

**Step 2: Check for a valid assignment of truth values**
* To make specification (2) true ($\neg p$), **$p$ must be False**.
* Now substitute $p = \text{False}$ into specification (1), which is $p \lor q$. For a disjunction to be true when the first part is false, **$q$ must be True**.
* Finally, test these values ($p = \text{False}$, $q = \text{True}$) in specification (3): $p \rightarrow q$ becomes $\text{False} \rightarrow \text{True}$. An implication with a false hypothesis evaluates to **True**.

**Conclusion:** Because the assignment **$p = \text{False}$ and $q = \text{True}$** makes all three requirements true at the same time, the system specifications are **consistent**.

---

#### EXAMPLE 5: Retransmission Buffer Inconsistency

> Do the system specifications from Example 4 remain consistent if we add a fourth specification: “The diagnostic message is not retransmitted”?

1. This new specification translates directly to **$\neg q$**.
2. From our analysis in Example 4, we established that the first three rules can *only* be simultaneously true when $p$ is false and $q$ is true.
3. However, if $q$ must be true, then this new requirement ($\neg q$) automatically evaluates to **False**.

**Conclusion:** There is no possible scenario where all four statements can be true at once. Therefore, these specifications are **inconsistent**.
