# Integer Algorithms and Modular Exponentiation (Section 4.2.3 - 4.2.4)

---

To evaluate how computers execute arithmetic, we measure the computational complexity of integer operations in terms of the total number of **bit operations** required to process $n$-bit numbers. This enables us to design highly efficient algorithms like **modular exponentiation**—a cornerstone of modern cryptography.

---

### 1. Positional Integer Algorithms

#### A. Binary Addition Algorithm
Given two binary numbers $a = (a_{n-1}\dots a_0)_2$ and $b = (b_{n-1}\dots b_0)_2$, we compute their sum bit by bit from right to left, adding the bits alongside a moving carry bit $c_j$.

```
procedure add(a, b: positive integers)
    c := 0
    for j := 0 to n - 1
        d := ⌊(a_j + b_j + c) / 2⌋   (determines the next carry)
        s_j := a_j + b_j + c - 2d    (determines the current sum bit)
        c := d
    s_n := c
    return (s_0, s_1, ..., s_n)
```

* **Complexity:** For two $n$-bit integers, the loop executes $n$ times. Each bitwise addition stage requires a constant number of operations. Thus, the computational complexity of binary addition is **$O(n)$** bit operations.

#### B. Binary Multiplication Algorithm
Binary multiplication mimics paper-and-pencil multiplication but leverages the fact that multiplying by a single binary bit ($b_j$) yields either the original value $a$ (if $b_j = 1$) or a flat $0$ (if $b_j = 0$). Every time we step left to the next digit, we multiply by a power of 2, which shifts the bit string one position leftward and pads the tail with a zero.

* **Complexity:** 
  * **Shifting:** Calculating partial products requires shifting strings up to $n$ places. Summing the shifts ($0 + 1 + 2 + \dots + n-1$) requires **$O(n^2)$** bit shifts.
  * **Addition:** We add $n$ partial lines together. Because each individual addition takes $O(n)$ bit additions, doing this across all lines results in a complexity of **$O(n^2)$** bit additions.
  * Therefore, standard multiplication has a baseline complexity of **$O(n^2)$** bit operations.

#### C. Algorithm for `div` and `mod`
To compute the quotient ($q = a \text{ div } d$) and remainder ($r = a \text{ mod } d$) programmatically using a basic structural subtraction loop:

```
procedure division_algorithm(a: integer, d: positive integer)
    q := 0
    r := |a|
    while r ≥ d
        r := r - d
        q := q + 1
    if a < 0 and r > 0 then
        r := d - r
        q := -(q + 1)
    return (q, r)
```

* **Complexity:** This brute-force method loop scales directly with the magnitude of the quotient, taking $O(q \log a)$ bit operations. Advanced hardware division implementations complete this process in **$O(n^2)$** bit operations for $n$-bit integers.

### 2. Modular Exponentiation

In cryptography (such as RSA encryption), algorithms regularly calculate $b^n \text{ mod } m$ where $b, n,$ and $m$ are massive integers containing hundreds of digits.

Calculating $b^n$ directly before applying the modulo operation is completely impossible for a computer since the value $b^n$ would exceed the memory capacity of any system. To solve this, we use the **repeated squaring algorithm**, which processes values using the **binary expansion of the exponent $n$**:
$$n = (a_{k-1}\dots a_1a_0)_2 \implies b^n = \prod_{i=0}^{k-1} b^{a_i \cdot 2^i}$$
Instead of multiplying sequentially $n$ times, we repeatedly square the base ($b, b^2, b^4, b^8, \dots$) and apply the modulo reduction at *every intermediate step* to keep the running values small and bounded.

#### Pseudocode:
```
procedure modular_exponentiation(b, m: positive integers, n = (a_{k-1}a_{k-2}...a_1a_0)_2)
    x := 1
    power := b mod m
    for i := 0 to k - 1
        if a_i = 1 then
            x := (x · power) mod m         (multiplies if the current bit is active)
        power := (power · power) mod m    (squares the base term for the next bit position)
    return x
```

#### Detailed Trace Example: Compute $3^{644} \text{ mod } 645$
1. **Convert Exponent to Binary:** $644 = (1010000100)_2$.
2. **Initialization:** $x = 1$, $\text{power} = 3 \text{ mod } 645 = 3$.
3. **Loop Executions:** We check bits $a_i$ from right to left ($i = 0$ to $i = 9$).

| Loop Iteration ($i$) | Exponent Bit ($a_i$) | Value of $x$ (Updated only if $a_i=1$) | Value of $\text{power}$ (Squared every loop) |
| :--- | :--- | :--- | :--- |
| **$i = 0$** | $a_0 = 0$ | $1$ | $3^2 \text{ mod } 645 = \mathbf{9}$ |
| **$i = 1$** | $a_1 = 0$ | $1$ | $9^2 \text{ mod } 645 = \mathbf{81}$ |
| **$i = 2$** | $a_2 = 1$ | $(1 \cdot 81) \text{ mod } 645 = \mathbf{81}$ | $81^2 \text{ mod } 645 = 6561 \text{ mod } 645 = \mathbf{111}$ |
| **$i = 3$** | $a_3 = 0$ | $81$ | $111^2 \text{ mod } 645 = 12321 \text{ mod } 645 = \mathbf{66}$ |
| **$i = 4$** | $a_4 = 0$ | $81$ | $66^2 \text{ mod } 645 = 4356 \text{ mod } 645 = \mathbf{486}$ |
| **$i = 5$** | $a_5 = 0$ | $81$ | $486^2 \text{ mod } 645 = 236196 \text{ mod } 645 = \mathbf{126}$ |
| **$i = 6$** | $a_6 = 0$ | $81$ | $126^2 \text{ mod } 645 = 15876 \text{ mod } 645 = \mathbf{396}$ |
| **$i = 7$** | $a_7 = 1$ | $(81 \cdot 396) \text{ mod } 645 = 32076 \text{ mod } 645 = \mathbf{471}$ | $396^2 \text{ mod } 645 = 156816 \text{ mod } 645 = \mathbf{81}$ |
| **$i = 8$** | $a_8 = 0$ | $471$ | $81^2 \text{ mod } 645 = 6561 \text{ mod } 645 = \mathbf{111}$ |
| **$i = 9$** | $a_9 = 1$ | $(471 \cdot 111) \text{ mod } 645 = 52281 \text{ mod } 645 = \mathbf{36}$ | (Loops terminate) |

* **Final Output Result:** $36$.

#### Complexity Analysis:
Instead of completing hundreds of sequential calculations, this squaring algorithm reduces the process to the length of the binary exponent string ($k \approx \log_2 n$). Consequently, fast modular exponentiation requires only **$O((\log m)^2 \log n)$** bit operations, making it highly efficient.

1. Trace the calculation of $2^9 \text{ mod } 11$ using the repeated squaring method. (Hint: Exponent $9 = 1001_2$).

---

### 💡 Solutions & Explanation

> [!NOTE]
> Here are the step-by-step calculations:
> 
> * **Initialization:** $x = 1$, $\text{power} = 2 \text{ mod } 11 = 2$.
> * **Exponent bits ($a_0$ to $a_3$):** $a_0 = 1, a_1 = 0, a_2 = 0, a_3 = 1$.
> * **Loop Tracing:**
>   * **$i = 0$ ($a_0 = 1$):**
>     * $x := (1 \cdot 2) \text{ mod } 11 = 2$
>     * $\text{power} := (2 \cdot 2) \text{ mod } 11 = 4$
>   * **$i = 1$ ($a_1 = 0$):**
>     * $x$ remains $2$
>     * $\text{power} := (4 \cdot 4) \text{ mod } 11 = 16 \text{ mod } 11 = 5$
>   * **$i = 2$ ($a_2 = 0$):**
>     * $x$ remains $2$
>     * $\text{power} := (5 \cdot 5) \text{ mod } 11 = 25 \text{ mod } 11 = 3$
>   * **$i = 3$ ($a_3 = 1$):**
>     * $x := (2 \cdot 3) \text{ mod } 11 = 6$
>     * $\text{power} := (3 \cdot 3) \text{ mod } 11 = 9$
> * **Final Output:** $6$.
>   *(Checking directly: $2^9 = 512$. $512 = 11 \cdot 46 + 6$, so $512 \bmod 11 = 6$. The algorithm is correct!)*
